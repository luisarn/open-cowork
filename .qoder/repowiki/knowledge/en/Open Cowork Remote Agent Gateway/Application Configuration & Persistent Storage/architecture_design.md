- **ConfigStore layer** (`config-store.ts`): Central singleton backed by `electron-store` with AES-256-GCM encryption and key rotation; manages `AppConfig` schema including provider selection, model presets, theme, sandbox mode, memory runtime settings, and up to 20 named config sets (profiles) each containing per-provider API keys, base URLs, and model IDs.
- **Auth utilities layer** (`auth-utils.ts`): Pure functions for credential resolution, base URL normalization (OpenAI-compatible, Anthropic, Ollama, Gemini), OAuth token detection, loopback detection, and empty-key allowance policies for local proxies.
- **Ollama integration layer** (`ollama-api.ts`): Fetches model index from `/models` endpoint with SHA-256-hashed cache keys, TTL-based caching, inflight deduplication, and connection testing via `/chat/completions`; also probes native `/api/show` for context window metadata.
- **Diagnostics engine** (`api-diagnostics.ts`): Sequential five-step pipeline (DNS → TCP → TLS → Auth → Model) using Node.js `dns`, `net`, `tls` modules plus OpenAI/Anthropic SDK clients; supports fast vs deep verification levels and produces structured fix recommendations.
- **Permission rules cache** (`permission-rules-store.ts`): In-memory mirror of renderer-originated tool permission rules with session-scoped "always allow" memory; sanitizes untrusted IPC payloads with conservative `'ask'` fallbacks.
- **SQLite persistence layer** (`database.ts`): Singleton `better-sqlite3` instance stored in `app.getPath('userData')/data/cowork.db` with WAL mode; manages sessions, messages, trace steps, and scheduled tasks tables with prepared statements and dynamic update builders; includes legacy file recovery and directory conflict handling.
- **Test routing shim** (`config-test-routing.ts`): Thin delegation to `probeWithClaudeSdk` for live API connectivity tests.

Dependency direction: All config-layer modules are pure or depend only on shared utilities (`logger`, `store-encryption`, `network/loopback`, `ollama-base-url`). The database module is independent of config-store. Both layers are consumed by the main process entry point (not in scope).
